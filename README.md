Принцип функционирования платформы
Архитектура веб-платформы построена на базе фреймворка Django и реляционной базы данных MySQL. Взаимодействие между компонентами реализуется по паттерновой схеме Model-View-Template (MVT).

Доступ и роли
Права разграничены на бэкенде. При каждом запросе декоратор @check_access проверяет текущую сессию пользователя (request.user) и его роль из таблицы Role. Для гостей с профилем Guest или email guest@forum.local система заменяет кликабельные элементы в шаблоне base.html на неактивные строки. Это блокирует отправку запросов на создание контента еще на этапе рендеринга.

Создание постов
Процесс сохранения топиков работает через разделение GET и POST методов в контроллере create_post_view. Переход по адресу /new-post/ вызывает GET-запрос и возвращает форму create_post.html. Отправка данных генерирует POST-запрос с параметрами title и content. Контроллер проверяет строки на пустоту и вызывает метод PostFromUser.objects.create(). Данные уходят в SQL-запрос INSERT INTO posts_from_users. Поля owner_id и временная метка created_at заполняются автоматически. После записи бэкенд выполняет редирект со статусом 302 на главную страницу.

Поиск и фильтрация
Поисковой движок на главной странице принимает параметры через GET-запрос на URL / с ключами search_topic и search_tag. Контроллер forum_home_view считывает строки, очищает их от пробелов и знаков #, после чего формирует динамический QuerySet. Фильтрация текстовых полей в MySQL выполняется с помощью объектов Q и SQL-оператора LIKE через модификатор __icontains. Это обеспечивает регистронезависимый поиск по вхождению слов одновременно в заголовки и текст публикаций. Введенный текст передается обратно в контекст шаблона, чтобы сохранить его в инпутах после обновления страницы.

Просмотр и детализация
Вывод данных изолирован для личного кабинета (profile_view) и страницы чтения поста (post_detail_view). В профиле бэкенд выполняет SQL-выборку с условием WHERE owner_id = request.user.id, подсчитывает количество записей через .count() и отдает массив карточек текущей сессии. Страница детального просмотра использует динамический роутинг /post/<int:post_id>/. Переданный идентификатор обрабатывается функцией get_object_or_404. Бэкенд делает точечный запрос к базе данных, извлекает текст статьи и данные автора, изолируя просмотр конкретной темы от общего пула
